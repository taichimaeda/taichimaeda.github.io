Apr 02, 20235 min read

Processes

Process allows a single processor to run multiple programs simultaneously.

  • Provide concurrency.
  • Provide isolation between programs.
    • Each process has its own address space.
  • Simplicity of programming.
    • No need to worry about behaviour of other programs.
  • Allow better utilisation of resources.
    • Different processes require different resources at certain times.

Types of Concurrency

  • Pseudo concurrency
    • Single processor is switched between processes by interleaving
    • Gives the illusion of concurrent execution over time.
  • Real concurrency
    • Utilises multiple physical processors.
    • Usually fewer processors than processes.

Context Switch

On a context switch, the processor switches from executing process to another process.

  • OS may switch processes in response to events or interrupts.
  • OS switches between processes cannot be pre-determined.
    • Because the events causing switches are non-deterministic.

It is important to avoid unnecessary context switches as they are expensive:

  • Direct cost
    • Save process state.
    • Restore process state.
  • Indirect cost
    • Perturbation of memory cache (TLB).
      • Typically flushed on a context switch.

Process Control Block

Information concerning the process is stored in a process descriptor or process control block (PCB):

  • Program counter
  • Stack pointer
  • Page table register
  • Process management information
    • Process ID
    • Parent process
    • Priority
  • File management information
    • Root directory
    • Current directory
    • Open file descriptors

Process Creation

Processes are created on:

  • System initialisation

  • User request

  • System call by a running process

  • Foreground process

    • Interacts with users.

  • Background process

    • Handles background jobs.
    • Known as daemons.

Process Termination

  • Normal completion
    • Process completes execution of body.
  • System call
    * e.g. exit() in UNIX and ExitProcess() in Windows.
  • Abnormal exit
    • Process has run into an error.
  • Aborted
    • Process interrupted by another process.
    • e.g. killed from terminal
  • Never
    • Some processes run in endless loop
    • e.g. server applications

Process Hierarchies

UNIX allows processes to create process hierarchies:

  • When UNIX boots, it starts running init.
  • init forks off one process per terminal.
  • Forked processes wait for some user to login.
  • When login is successful, the login process runs a shell process to accept commands.
  • All processes in the system form a tree with init as root (i.e. process group).

Windows has no notion of hierarchy:

  • When a child process is created, the parent is given a handle for its control.
  • Handle can be passed to another processes.
  • Need to specify all parameters on creation.

Forking Process

Linux fork() creates new child process by making exact copy of parent process image.

  • Child process inherits resources of parent process.
  • Executed concurrently with parent process.
  • In parent process, returns PID of child.
  • In child process, returns 0.
  • On error, no child is created and returns -1.

Executing Process

Linux execve() runs the executable at the given path in a new process.

  • path specifies the full path of executable.
  • arg specifies the arguments passed to main().
  • envp specifies the environment variables.

Waiting for Process Termination

Linux waitpid() suspends execution of calling process until process with the given PID terminates.

  • PID of -1 tells the process to wait for any child.
  • PID of 0 tells the process to wait for any child in the same process group.
  • PID of -gid tells the process to wait for any child with the group id gid.
  • Returns PID of the terminated process.
  • Returns 0 if no children terminated.
  • Returns -1 on error.

Terminating Process

Linux exit() terminates a process.

  • Called implicitly when program finishes execution
  • Never returns in the calling process
    • Returns an exit status to the parent process

Linux kill() terminates a process and sends signal to process with the given PID.

Unix Signals

Unix signals are inter-process communication mechanism used to notify processes when event occurs.

Signals are generated:

  • When exception occurs.
    • e.g. division by zero (SIGFPE), segment violation (SIGSEGV)
  • When kernel wants to notify the process.
    • e.g. process writes to a closed pipe (SIGPIPE)
  • When certain key combinations are typed in a terminal.
    • e.g. keyboard interrupt (SIGINT)
  • Programatically by kill() system call.

Sending a signal requires a special permission (e.g. same user or super user). Kernel can send signals to any process.

Default action for most signals is to terminate the process, but the receiving process may choose to:

  • Handle the signal by installing a signal handler with signal().
  • Ignore the signal.
    • SIGKILL and SIGSTOP cannot be ignored.

Unix Pipes

Unix pipe is a method of connecting the standard output of one process to the standard input of another.

  • Allows for one-way communication between processes.
  • Widely-used on the command line and in shell scripts.

Linux pipe() returns two file descriptors in fd:

  • fd[0]: read end of the pipe

    • Sender should close the unused read end.

  • fd[1]: write end of the pipe

    • Receiver should close the unused write end.

  • If receiver reads from empty pipe, it blocks until data is written at the other end.

  • If sender attempts to write to full pipe, it blocks until data is read at the other end.

Persistent pipes are stored on file system and can outlive process which created them. Any process can open persistent pipes like regular files.

Process has its own virtual machine

  • Own virtual CPU
  • Own address space (stack, heap, text, data etc.)
  • Open file descriptors
