File System Functionalities

  • Logical name to physical disk address translation
  • Management of disk space
    • Allocation and deallocation
  • File locking for exclusive access
  • Performance optimisation
    • Caching and buffering
  • Protection against system failure
    • Back-up and restore
  • Protection against unauthorised access

File Attributes

File attributes may be held within given directory system.

  • Basic information
    • File name
    • File type
    • File organisation
    • File creator
  • Address information
    • Volume
    • Start address
    • Size used
    • Size allocated
  • Access control information
    • Owner (often creator)
    • Authentication
    • Permitted actions
  • Usage information
    • Creation timestamp
    • Last modified
    • Last read
    • Last archived
    • Expiry date
    • Access activity counts

Space Allocation

Space is allocated in blocks of 512-8192 bytes:

  • Block size too large
    • Wastes space for small file.
      • Unused space in each block.
    • More memory needed for buffer space.
  • Block size too small
    • Wastes space for large files.
      • Meta data in each block.
    • High overhead in terms of block management.
    • High file transfer time.
      • Seek time is greater than transfer time.

Dynamic space management is necessary as file sizes are naturally variable.

Various methods exist for accessing blocks:

  • Contiguous file allocation
  • Block chaining
  • File allocation table
  • Index blocks

Contiguous File Allocation

Place file data at contiguous addresses on storage device.

  • Advantages
    • Easy to map logical address.
  • Disadvantages
    • External fragmentation
    • Poor performance if file grows over time
      • Must be transferred to new area of adequate size
      • Leads to additional IO operations

Block Linkage

Place file data in a linked list.

  • List elements are stored on disk.

    • Each element has a pointer to the next element.

  • Wastes pointer space in each block.

  • Large block sizes

    • Can result in internal fragmentation

  • Small block sizes

    • Data spread across multiple blocks
    • Poor performance due to many seeks

Block Allocation Table

Use a separate table to store the links between blocks.

  • Table entries are stored on disk.
    • Each key points to the current block.
    • Each value points to the next block.
  • Reduces the number of seeks
    • No need to read an entire block to find the next element.
      • Only read blocks containing the table.
    • Less effective for large tables with spread entries.
      • May need to read BAT block many times.
      • Occurs due to file fragmentation.
      • Periodic defragmentation to bring blocks close together.

Index Blocks

  • Each file has one or more index blocks
    • Contains list of pointers to data blocks.
    • Directory points to the file’s index block.
    • Index block may be chained by another index block.
  • Place index blocks near the corresponding data blocks for quick access.

In Unix, index blocks are called inodes.

  • Inode table maps inode number to inode.
  • Inode has the same structure as index block but includes:
    • Disk device number
    • Reference count
    • Major/minor device numbers
    • Inode number
      • Allows the file system to identify inode for rename/move.
    • Direct/indirect pointers
      • Provides fast access to small files while supporting very large files.

File system maintains a tree of file/directories names and the corresponding inode numbers.

Free Space Management

Free List

Use free list for quick access to free blocks for allocation:

  • Linked list of blocks containing locations of free blocks.
    • Blocks are allocated from the beginning of free list.
    • Newly freed blocks are appended to end of list.
  • Low overhead to perform free list maintenance operations.
  • Files are likely to be allocated in non-contiguous blocks.
    • Increases file access time.

Bitmap

Bitmap contains one bit (in memory) for each disk block

  • Indicates whether block is in use.
    • th bit set corresponds to th block used.
  • Can quickly determine available contiguous blocks.
  • May need to search entire bitmap to find free block.

File System Layout

  • Super block
    • Number of inodes

    • Block size
    • Maximum file size

File System Directory Organisation

Directory maps symbolic file names to logical disk locations (e.g. wombat.txt to disk 0, block 2).

  • Helps with file organisation.
  • Ensures uniqueness of names.

Hierarchical file system (e.g. UNIX, Linux, Windows, Mac, …):

  • Root indicates where on disk root directory begins.
  • Root directory points to various directories.
    • Each of which contains entries for its files and subdirectories.
  • File names need be unique only within given directory.
  • File names are given by:
    • Path from root directory (absolute)
    • Path from current directory (relative)

Directory Operations

  • Search directory
  • List files/subdirectories
  • Create/delete file/directory
  • Open/close directory
  • Link/Unlink
    • Create/remove link to file/directory.
  • Mount/Unmount
    • Create/remove link to file/directory in a different file system.
    • e.g. another disk or remote server

Linking

  • Hard link
    • References address of file.
    • Only supported for files in Unix.
  • Soft link
    • References full pathname of file.
    • Created as directory entry

There are a few problems with linking:

  • Deleting linked files.
    • Keep link count with file (hard links)
    • Leave links and cause exception when used (soft links)
  • Directory traversal algorithms may loop.

Mounting

  • Combines multiple file systems into one namespace.
    • Allows reference from single root directory.
  • Supports soft-links to files in mounted file systems (not hard-links)

File systems manage mounted directories with mount table, which stores information about location of mount points and devices.
