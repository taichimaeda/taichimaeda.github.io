Disk Management

Disk Addressing

Disks are divided into tracks and sectors.

  • Physical hardware address is complicated (e.g. cylinder, surface, sector).
  • Modern disks use logical sector addressing.
    • Sectors numbered consecutively.
    • Simplifies disk management.
    • Helps work around BIOS limitations on addressing.

Disk Formatting

Disks must be formatted before use.

  • Low-Level Formatting
    • Divides the disk into sectors and tracks.
    • Creates the inter-sector gaps and sector headers
    • Typically done at the factory during the manufacturing process.
      • Complex and time-consuming.
      • Potentially damages the disk if not done correctly.
  • High-Level Formatting
    • Prepares the disk for use by the operating system.
    • Creates a file system on the disk.
    • Assigns logical addresses to the sectors and tracks.
    • More commonly used in modern systems.

Disk Cacheing

Disk-caching aims to use main memory to improve disk access.

  • Read-ahead
    • Caches data on the disk controller before it is requested.
    • Improves read performance.
  • Write-back caching
    • Caches write operations in memory instead of immediately writing to disk.
    • Can introduce data consistency if the system crashes before the data is written to the disk.
    • Improves write performance.

Least Recently Used Replacement

Replace block that was in cache longest with no references.

  • Cache consists of stack of blocks.
    • Most recently referenced block on top of stack.
    • When block is referenced, place on top of stack.
    • Remove block at bottom of stack when new block brought in

Least Frequently Used Replacement

Replace block that has experienced fewest references.

  • Counter associated with each block.
    • Counter incremented each time block accessed.
    • Block with the smallest count is selected for replacement.
  • Some blocks may be referenced many times only in a short period of time.
    • Leads to misleading reference count.

Frequency Based Replacement

  • Divide LRU stack into new and old sections
    • When block is referenced, place on top of stack.
    • Only increment reference count if in the old section.
    • Block with the smallest count is selected for replacement.
  • Some blocks may be replaced too early if referenced in the new section many times.
    • Use three sections and only replace blocks from the old section.

Disk Performance

  • Seek time

    • Time it takes to move to the correct track.
    • Longer than rotation time.
    • More important to optimise.
    • Negligible when moving to the next track.

  • Rotation time

    • Time it takes to rotate to the correct sector on track.

  • Transfer time

    • Time it takes to read/write data once the read-write head is in the correct position.

Disk Scheduling

FCFS Scheduling

  • No ordering of requests
    • Random seek patterns
  • Suitable for lightly-loaded disks
  • Poor performance for heavy loads
  • Fair scheduling

SCAN Scheduling

  • Choose requests which result in shortest seek time in preferred direction.
  • Only change direction when:
    • Reaching outermost/innermost track
    • There are no further requests in preferred direction.
  • Minimises the movement of the read/write head.
  • Most common scheduling algorithm
  • Long delays for requests at extreme locations

IO Request

  • IO requests are placed in request list.
    • One request list for each device in system.
    • bio structure associates memory pages with requests.
  • Block device drivers define request operation.
    • Kernel passes ordered request list
    • Driver must perform all operations in list
  • Some device drivers (e.g. RAID) order their own requests.

IO Request Merging

In IO request merging, the kernel attempts to merge read/write requests to adjacent blocks.

  • Synchronous read requests may starve during large writes.
    • Asynchronous read requests can wait.
    • Write requests are usually asynchronous.
  • Deadline scheduler
    • Ensures reads performed by deadline.
    • Eliminates read request starvation.
  • Anticipatory scheduler
    • Reorders IO request list by anticipating the future requests.
    • Anticipation is based on requests that arrived during waiting period.
    • Reduces excessive seeking behaviour.
    • Can lead to reduced throughput if anticipation fails.
