Device Management

IO Management

Objectives of IO device management:

  • Fair access to shared devices.
    • Allocation of dedicated devices.
  • Exploit parallelism of I/O devices for multiprogramming.
  • Provide uniform simple view of I/O.
    • Hide complexity of device handling.
    • Give uniform naming and error handling

Device Classes

Devices are grouped into device classes:

  • Members of each device class perform similar functions.
  • Allows kernel to address performance needs of certain device class.

Major and minor identification numbers:

  • Used by device drivers to identify their devices.
  • Devices with same major number are controlled by same driver.
  • Minor numbers enable system to distinguish between devices of same class.

Device Independence

  • Device independence from
    • Device type (e.g. terminal, disk or DVD drive)
    • Device instance (e.g. which disk)
    • Device variations
      • Unit of data transfer (e.g. character or buffered block)
      • Supported operations (e.g. read, write, seek)
      • Synchronous or asynchronous operation
      • Speed differences
      • Sharable or single user (e.g. printer, DVD-RW)
      • Types of error conditions

Device Independent Layer

Device independent layer provides device independence:

  • Maps logical to physical devices (naming and switching).
  • Requests validation against device characteristics.
  • Allocation of dedicated devices.
  • User access validation.
  • Buffering for performance and block size independence.
  • Error reporting.

Device Driver

Device driver handles one device type, but may control multiple devices of same type:

  • Implements block read or write
  • Access device registers
  • Initiate operations
  • Schedule requests
  • Handle errors

Interrupt Handler

  • Device sends an interrupt to notify the process when operation is complete.
  • Interrupt handler processes these interrupts.
    • For block devices:
      • On transfer completion, signal device handler.
    • For character devices:
      • On character transfer, process next character.

Dedicated, Shared and Spooled Devices:

  • Dedicated device (e.g. DVD writer, terminal, printer)
    • Dedicated to a single process.
    • Open fails if already opened.
      • Alternatively, queue open requests.
    • Allocated for long periods.
    • Allocated only to authorised processes.
  • Shared device (e.g. disks, window terminals)
    • Shared by multiple processes.
    • OS provides file system for disks.
  • Spooled devices (e.g. printers)
    • Content is saved to disk file.
    • File printed later by spooler daemon.
      • Printer only allocated to spooler daemon.
      • No normal process allowed direct access.
    • Avoids blocking access to non-shareable devices.

Buffered, Unbuffered IO

  • Buffered IO:
    • Used to smooth peaks in IO traffic.
    • Caters for differences in data transfer units between devices.
    • For output:
      • User data transferred to OS output buffer.
      • Process continues and only suspends when buffer full.
    • For input:
      • OS reads ahead; reads normally satisfied from buffer.
      • Process blocks when buffer empty.
  • Unbuffered IO:
    • Data is transferred directly from user space to/from device.
    • Each read/write causes physical IO.
      • Implies device handler used for each transfer.
      • High process switching overhead.

User-Level IO Interface

  • IO operations
    • e.g. open, close, read, write, seek
  • OS library procedures to set up IO parameters
    • Must be device independent
    • Synchronous or asynchronous
    • Blocking or non-blocking

Unix allows user to access virtual devices as files.

Linux IO Management

Loadable Kernel Module

Linux’s LKM provides device drivers:

  • Contain object code, loaded on-demand
    • Dynamically linked to running kernel
    • Provided by hardware vendors or independent developers
  • Require binary compatibility
    • Modules written for different kernel versions may not work

Kmod is a kernel subsystem that manages modules without user intervention:

  • Determines module dependencies.
  • Loads modules on demand.

Device Special Files

Most devices in Linux are represented by device special files.

  • Entries in /dev directory provide access to devices.
  • List of devices in system can be obtained by reading contents of /proc/devices.

Virtual File System

Device files accessed via virtual file system:

  • System calls pass to VFS, which in turn issues calls to device drivers.
  • Most drivers implement common file operations.
    • e.g. read, write, seek

Linux IO Classes

  • Character (unstructured): Files and devices
  • Block (structured): Devices
  • Pipes (message): Interprocess communication
  • Socket (message): Network interface

Linux Character Device

  • Transmits data as stream of bytes.
  • Represented by char_device_struct structure which contains:
    • Driver name.
    • Registered major and minor numbers.
    • Pointer to driver’s file_operations structure.
  • All registered drivers referenced by chrdevs vector.

Linux Block Device

Kernel’s block IO subsystem contains a number of layers, which modularises operations by placing common code in each layer.

There are two primary strategies used by kernel to minimise overhead of accessing block devices:

  • Caching data
    • Search cache when data from block device is requested.
    • If found, data is copied to process’s address space.
    • Otherwise, data is added to request queue.
    • Use direct IO to bypass kernel cache:
      • Important for databases and other applications
      • Kernel caching inappropriate and may reduce performance/consistency
  • Clustering IO operations

Linux Socket Device

  • Allows bidirectional communication.
  • Can be used to exchange information both locally and across a network.
    • Unlike pipes which are identified by machine specific file descriptors
  • Two types of sockets:
    • TCP (stream sockets)
    • UDP (datagram sockets)

File Descriptor Table

Each process has its own file descriptor table.

  • Each process has three file descriptors when created:

    • stdin (0)
    • stdout (1)
    • stderr (2)

  • By default, all three file descriptors refer to terminal from which program was started.

  • Blocking IO

    • IO call returns when operation completed
    • Process is suspended until the operation is completed.
    • Easy to understand but leads to multi-threaded code.
      • Prevent the program from being unresponsive.

  • Non-blocking IO

    • IO call returns as much as available (e.g. read with 0 bytes).
    • Turn on for file descriptor using fcntl system call.
    • Provides application-level polling for IO.
      • Application periodically checks to see if more data is available.

  • Asynchronous IO

    • Process executes in parallel with IO operation.
      • No blocking in interface procedure.
    • IO subsystems notifies process upon completion via callbacks or process signals.
    • Very flexible and efficient but harder to use and potentially less secure.
      • Timing and order of execution can be exploited.
