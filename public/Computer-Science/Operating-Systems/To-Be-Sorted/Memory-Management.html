Memory Management

Memory management binds logical address space to physical address space.

  • Logical address

    • Generated by the CPU.
    • Address space seen by process.

  • Physical address

    • Address seen by the memory unit.
    • Refers to physical system memory.

  • Compile-time and load-time address-binding scheme:

    • Logical address and physical address are the same.

  • Execution-time address-binding scheme:

    • Logical address and physical address are different.

Note that memory address is given in units of byte. Two hexadecimal digits represents a byte.

Memory Management Unit

MMU is a hardware device for mapping logical to physical addresses

  • User process deals with logical addresses only.
  • Implemented in hardware for speed.
  • e.g. Add value in relocation register to every address generated by process

Main memory usually split into two partitions:

  • Kernel space
    • Used by resident operating system.
    • Held in low memory with interrupt vector.
  • User space
    • Held in high memory.

Memory Allocation

Contiguous Memory Allocation

Contiguous allocation is done with relocation registers:

  • Base register contains value of smallest physical address for each process.
  • Limit register contains range of logical addresses for each process.
  • MMU maps logical address dynamically.

Multiple-Partition Allocation

A hole is a block of available memory. Holes of various size are often scattered throughout memory.

When a new process arrives, OS must allocate memory from hole large enough. For this purpose OS maintains information about:

  • Allocated partitions.
  • Free partitions (hole).

Dynamic Storage Allocation

Strategies for how to satisfy request of a certain size from list of free holes:

  • First-fit: Allocate first hole that is big enough
  • Best-fit: Allocate smallest hole that is big enough
    • Must search the entire list, unless ordered by size.
    • Produces smallest leftover hole.
  • Worst-fit: Allocate largest hole
    • Must search the entire list
    • Produces largest leftover hole.

Fragmentation

  • External fragmentation
    • There is enough free memory but not contiguous.
  • Internal fragmentation
    • Allocated memory larger than requested memory.
    • Due to size difference internal to partition.

External fragmentation can be reduced by compaction:

  • Rearrange memory to place all free memory together in one large block.
  • Leads to I/O bottlenecks.

Memory Protection

Check memory access is valid to protect user processes.

  • From each other.
  • From changing operating-system code and data.

Memory protection is implemented by associating protection bits with pages:

  • Valid-bit indicates legal page:
    • Associated page exists in logical address space.
  • Invalid-bit indicates page missing:
    • Associated page does not exit logical address space (page fault).
    • Need to load page from disk (c.f. demand paging).
    • Could be an incorrect access.

Virtual Memory

Ideas of virtual memory:

  • Separation of user logical memory from physical memory.
    • Only part of process needs to be in memory for execution.
    • Logical address space can be much larger than physical address space.
    • Address spaces can be shared by several processes.

Virtual memory can be implemented via:

  • Segmentation
  • Paging

Segmentation

Segmentation uses special registers for indicating the base address of memory for each process, which get updated on context switch:

  • Address space that can grow/shrink independently.
  • Support different kinds of protection (read/write/execute).
  • Unlike pages, programmers are aware of segments.

Segmentation provides separate address spaces for code, data, stack.

  • Code segment (CC)
  • Data segment (DS)
  • Extra segment (ES)
  • Stack segment (SS)

Paging

Logical address space of process can be noncontiguous in physical address. Process gets allocated physical memory when available.

  • Frames
    • Fixed-sized blocks of physical memory.
    • OS keeps track of all free frames.
  • Pages
    • Block of same size of logical memory.

To run program of size pages:

  1. Find free frames and load program.
  2. Set up page-table to translate logical to physical addresses.
    • Page-table is stored in kernel memory per process.
    • Page-table base register (PTBR) points to page table.
    • Page-table length register (PTLR) indicates size.
    • Each page contains address of its frame in physical memory.

User space address is divided into:

  • Page number ()
    • Used as index into page table.
  • Page offset ()
    • Combined with base address.

Kernel space address is simply mapped by offset, as this makes it easy to detect user access to kernel space.

Fixed-size paging avoids external memory fragmentation.

Page table stores an entry for every virtual page and can be extremely large. To address this issue we can use:

  • Hierarchical paging
  • Hashed page table
  • Inverted page table

Translation Look-aside Buffers

In paging, every data/instruction access requires two memory accesses.

TLB uses special fast-lookup hardware cache as associative memory:

  • If in associative register, get frame address out.
  • Otherwise get frame address from page-table in memory.

TLB needs to be flushed after context switch, which can lead to substantial overhead.

  • Some TLBs store address-space ids in entries
    • Identifies each process to provide address-space protection for that process.
    • Allows kernel pages to be remain in TLB.

Hierarchical Paging

Hierarchical paging breaks up logical address space into multiple page tables.

  • Two-level page table
    • First page table
      • Page number: 12 bits
      • Page offset: 20 bits
    • Second page table
      • Page number: 10 bits
      • Page offset: 10 bits

Hashed Page Table

  • Store entry based on hashed page numbers.
    • Hashed page number is the key.
    • Frame number is the value.
  • Search for match of hashed page number in key.
  • Use separate chaining for page numbers with the same hash.

Inverted Page Table

  • Store entry for all frame numbers.
    • Frame number is the key.
    • Page number is the value.
  • Search for match of page number in value.
  • Increases time to search table when page reference occurs.
    • Can use hash table to limit search.

Swapping

  • Swap processes temporarily out of memory to disk.
  • Bring back into memory for continued execution.
  • Swap space can be file or dedicated partition on disk.
  • Transfer time is the major part of swap time.

Use swapping when:

  • Number of processes is limited by amount of available memory
  • Only running processes need to be in memory.

Shared Memory

Processes can set up shared memory areas:

  • Implicitly or explicitly mapped to files on disk.
  • After shared memory is established, no need for kernel involvement.

Demand Paging

Demand paging brings page into memory only when needed:

  • Lower I/O load
  • Less memory needed
  • Faster response time
  • Support for more user

OS references a page on demand:

  • If entry not found, abort.
  • If valid bit set:
    • Page exists in memory.
    • Translate page number to frame number.
  • If valid bit not set:
    • Page does not exit in memory.
    • Page fault to bring page into memory.
      • Get empty frame.
      • Swap page into frame.
      • Set valid bit.
      • Restart last instruction.
    • Translate page number to frame number.

Copy on Write

Allows parent and child processes to initially share same pages in memory (e.g. fork()).

  • If either process modifies shared page, then copy page.
  • Efficient process creation as only modified pages are copied.
  • Free pages allocated from pool of zeroed-out pages.

fork() gives child its own page table pointing to parent’s read-only pages:

  • When any process writes to a page
    • Protection fault causes trap to the kernel.
    • Kernel allocates new copy of page so that both processes have their own private copies.
    • Both copies are marked read-write.

Memory Mapped Files

  • Map file into virtual address space using paging.
  • Simplifies programming model for I/O.

Page Replacement

Page replacement attempts to find some unused page in memory to swap out when there is no free frame available.

Strategies for efficient page replacement:

  • Minimise number of page faults.
    • Avoid bringing same page into memory several times.
    • Expects page faults to decrease with more frames.
  • Use dirty bit to reduce overhead of page transfers.
    • Only modified pages written to disk.

Reference string is a particular string of memory references, which is used to evaluate algorithms by computing number of page faults.

First-in-First-out Algorithm

FIFO algorithm replaces the oldest page in page table.

  • May replace heavily used page
  • Belady’s Anomaly
    • More frames can lead to more page faults.

Optimal Algorithm

Optimal algorithm replaces page that is unused for the longest period of time.

  • Impractical
  • Used for measuring the ideal performance.

Least-Recently-Used Algorithm

In LRU algorithm, each page entry has a counter:

  • When page is referenced, copy clock into counter.
  • When page needs to be replaced, choose the lowest counter.

Proper LRU is expensive, so approximations are used in practice:

Least Frequently Used Algorithm

LFU algorithm replaces page with the smallest access count.

  • May replace page just brought into memory.
  • Never forgets heavy page usage.
    • Reset counters periodically.
    • Use aging.

Most Frequently Used Algorithm

MFU algorithm replaces page with the largest access count.

  • Replaces frequently used page.
  • Never forgets unused pages.

Global and Local Replacement

  • Local strategy
    • Each process gets fixed allocation of physical memory.
    • Need to pick up changes in working set size.
  • Global strategy
    • Dynamically share memory between runnable processes.
    • Initially allocate memory proportional to process size.
    • Consider page fault frequency to tune allocation.
      • Measure page faults per second.
