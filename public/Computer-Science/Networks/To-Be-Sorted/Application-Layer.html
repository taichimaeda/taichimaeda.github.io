Application Layer

World Wide Web

WWW is based on the idea of hypertext and hyperlinks. It was extremely successful due to the following factors:

  • simplicity of HTML
  • simplicity of HTTP
  • low entry barrier for publishers
  • GUI browsers (e.g. Mosaic, Netescape)

An end system called a host may run multiple programs which run multiple processes. A process is addressed within its host by a port number.

Applications use the network through sockets, which is an interface provided by the operating system.

Architecture of a typical network communication:

  • Client application
    1. Create a socket by connecting to host on port .
    2. Read and write data using the socket .
    3. Disconnect by destroying .
  • Server application
    1. Create a socket by accepting a connection on host on port .
    2. Read and write data using the socket .
    3. Disconnect by destroy .

Some applications have processes that act both as clients and servers, often called Peer-to-Peer (P2P) architecture.

Hyper-Text Transfer Protocol

The main purpose of HTTP is to provide access to Web objects through a connection-oriented transport mechanism.

  • Consists of a sequence of requests issued by the client.
  • Responses issued by the server, each in response to a single request.
  • Stateless protocol.

HTTP runs on port 80 with TCP.

Example of an HTTP request:

GET /~kgk/212/index.html HTTP/1.1
Host: www.doc.ic.ac.uk
User-agent: Mozilla/5.0
Accept-Language: en-GB

Host is passed as a parameter so that server can host multiple virtual sites e.g. Dokku

Example of an HTTP response:

HTTP/1.1 200 OK
Date: Thu, 26 Jan 2017 16:00:00 GMT
Server: Apache
Last-Modified: Sat, 21 Jan 2017 20:05:55 GMT
Accept-Ranges: bytes
Content-Length: 5470
Content-Type: text/html
...
<!DOCTYPE html>
...

Telnet allows us to send plain-text commands directly to a server listening on a specific port.

Protocol Version of HTTP

A HTTP protocol should always include a version number.

The first version of HTTP used one TCP connection per object, which is both an inefficient use of the network and the operating system.

HTTP/1.1 introduced persistent connections, in which multiple requests and responses can be issued. Connection: close in the request and response asks the server/client to close the persistent connection.

HTTP/2 is expected to completely replace HTTP/1.x within the next few years. In HTTP/2, content is binary, connection is not ordered/blocking and thus capable of using a single (TCP) connection for parallelism.

HTTP Methods

  • GET: retrieves the object at the URL.
  • POST: submits the enclosed data to the server.
    • Not idempotent e.g. x = x + 1.
  • HEAD: only retrieves the response header of the object at the URL.
  • PUT: requests the enclosed data to be stored under the URL.
    • Idempotent e.g. x = 4.
  • DELETE: requests the object at the URL to be deleted.
  • OPTIONS: requests the available communication options for the object at the URL.

HTTP Headers

  • Object characterisation
    • e.g. Content-Type, Content-Length, Content-Encoding
  • Content negotiation
    • e.g. Accept-Charset, Accept-Encoding
  • Cache management
    • e.g. Expires, Last-Modified, ETag
  • Explicit cache control
    • e.g. Cache-Control
  • Method-specific responses
    • e.g. Allow
  • Authorisation/identification
    • e.g. Authorization

HTTP Status Codes

  • 1xx: Informational
  • 2xx: Successful operation
  • 3xx: Redirection
    • Moved permanently (301)
    • Found/Moved temporarily (302)
  • 4xx: Client error
    • Malformed request (400)
    • Unauthorised (401)
    • Object not found (404)
    • Method not allowed (405)
  • 5xx: Server error
    • Internal server error (500)
    • Service overloaded (503)

Structure of HTTP Request

  1. Request line
    1. Method (e.g. GET)
    2. Space
    3. URL (e.g. /~kgk/212/index.html)
    4. Space
    5. Protocol version (e.g. HTTP/1.1)
  2. Header lines (optional)
    1. Header name (e.g. Host)
    2. Colon
    3. Space
    4. Header value (e.g. www.doc.ic.ac.uk)
  3. Empty line
  4. Object body (optional)
  5. Line terminator (CRLF)

Structure of HTTP Response

  1. Status line
    1. Protocol version (e.g. HTTP/1.1)
    2. Space
    3. Status code (e.g. 405)
    4. Space
    5. Status message (e.g. Method Not Allowed)
  2. Header lines (optional)
    1. Header name (e.g. Date)
    2. Colon
    3. Space
    4. Header value (e.g. Thu, 26 Jan 2017 16:00:00 GMT)
  3. Empty line
  4. Object body (optional)
  5. Line terminator (CRLF)
    1. CR (\r) and LF (\n) both required for Windows and HTTP.
    2. Only LF (\n) required for Unix.

Web Caching

With web caching, a client request goes to a proxy/cache server.

The proxy may:

  • Forward the request to the origin server, acting as a client.
  • Retrieve the response from the origin server.
  • Cache the object for some time.
  • Forward the response back to the client.

The benefits of web caching include:

  • Reduced latency
  • Reduced network traffic
  • Privacy
  • Protection from outside access

Cache Control for Web Caching

  • HEAD request is used to obtain cache information about the object.
    • e.g. to see if object has been updated.
  • Servers specify explicit expiration times in HTTP headers.
  • Cache-Control: no-cache
    • Request: proxy should not return the cache.
    • Response: client should not cache this object
  • Cache-Control: max-age=X
    • Request: proxy should return the cache if less than X seconds old.
    • Response: client should keep the cache up to X seconds.
      • Should be used with must-revalidate otherwise the restriction is optional.
  • Expires: X
    • Request: proxy should return the cache if before X.
    • Response: client should keep the cache until X.
      • Should be used with must-revalidate otherwise the restriction is optional.
  • If-Modified-Since: X
    • Request: proxy should only return the cache if modified since X.

Sessions and Cookies

HTTP provides Cookies as the means for higher-level applications to maintain stateful sessions.

  • Request: Cookie header is used to tell the server that the request belongs to the given session.
  • Response: Set-Cookie header is used to tell the client to store the given cookie as a session identifier.

Dynamic Webpages

  • Common Gateway Interface (CGI)
    • URL specifies the program and its parameters.
    • Server starts a process to execute the program and return response.
    • Stateless.
  • Servlets
    • Java-based solution for CGI
    • Stateful.
  • Embedded scripts
    • Server-side: PHP
    • Client-side: JavaScript

IP Addresses and Host Names

An end system is identified and addressed by its IP address.

  • 32 bits in IPv4
  • 128 bits in IPv6

An IP address is easy to process for computers but are not practical for everyday use. Host names are used for this purpose.

The mapping from host names to ip addresses are resolved by:

  • /etc/hosts file
    • User-defined host names.
    • Used up to 1983 when there were not many hosts.
  • Domain Name System (DNS)
    • Distributed lookup facility.

Domain Name System

Primary function of the DNS is to map a name to an IP address.

The name space was divided into non-overlapping zones, each of which is handled by one or more name servers.

  • Root servers
    • Holds the map from the top-level domain to the associated NS.
  • Top-level domain servers
    • Holds the map from the second-level domain to the associated NS.
  • Authoritative server
    • Holds the map of public hosts within the given domain.

DNS runs on top of UDP because it can simply resend the request if no response is received.

Resource Records of DNS

DNS is essentially a database that contains Resource Records (RR) with the following columns:

There are several values available for Type column in RR:

  • Type A
    • Maps a host name to its IP address.
    • Name is the host name.
    • Value is the IP address.
  • Type NS
    • Maps a host name to its name server.
    • NS holds Type A/NS record for the host name.
    • Name is the host name.
    • Value is the NS host name.
  • Type CNAME
    • Maps a host name to its canonical host name.
    • Name is the host name (alias).
    • Value is the canonical host name.
    • e.g. www3.imperial.ac.uk to www.imperial.ac.uk
  • Type MX
    • Maps a host name to its mail exchange server .
    • Name is the host name.
    • Value is the host name for the MX server that handles mail for the given host name.

TTL column specifies the time to live, which tells how long the record should be valid for.

Round Robin DNS

Round robin DNS is a technique for balancing the load of geographically-distributed Web servers.

  • DNS server holds a mapping from host name to multiple name servers, one of which is returned in round robin order per request.
  • TTL should be small because these may constantly change depending on server load.

DNS Caching

The benefits of DNS caching include:

  • Improved performance

  • Reduced load on the DNS infrastructure

  • /etc/resolv.conf file contains a list of DNS cache servers

    • Typically only contains the router’s IP address.
    • The router queries the local ISP’s DNS cache server.
    • The local IPS’s DNS cache server returns the DNS cache, or queries the authoritative NS (e.g. root, top-level).

DNS cache poisoning is a type of cyber attack that exploits a vulnerability in the DNS to redirect users to fraudulent websites. This can be prevented by the authoritative NS signing RR etc.

Content Distribution Networks

Streaming content to a large number users of simultaneously:

  • Single large server
    • Single point of failure.
    • Point of network congestion.
    • Long path to distance users.
  • Content distributed networks
    • Stores multiple copies of videos at multiple geographically distributed sites.

CDN cluster design strategies:

  • Enter deep strategy
    • Push CDN servers deep into many access networks.
    • 10-100K servers.
  • Bring home strategy
    • Smaller number of larger clusters in PoPs near access networks.
    • 10-100 servers.

CDN cluster selection strategies:

  • Pick a CDN node geographically closest to client.
  • Pick a CDN node with shortest delay to client.
    • CDN nodes periodically ping local ISPs and report results to CDN DNS server.

CDN DNS server only knows the local DNS cache server, which may not be fully accurate:

  • Public DNS
    • When DNS cache is unavailable, queries CDN DNS server with the client’s actual IP.
      • Performs better redirection.
    • e.g. 8.8.8.8 by Google, 1.1.1.1 by CloudFlare.
  • Client approach
    • By Netflix
    • CDN DNS servers return a list of several CDN servers.
    • Client pings and picks the best server based on RTT.

Electronic Mail

Features and goals of emails:

  • Asynchronous communication
  • One-to-many communication
  • Multimedia content

Architecture of email systems:

  • User agent
    • Allows a user to read and write emails.
  • Mail transfer agent (mail sever)
    • Accept messages for remote delivery .
      • Store messages in a local persistent queue.
      • Deliver messages to the destination server using the transport protocol.
    • Accept messages for local delivery.
      • Save messages in some local persistent mailbox.
      • Allow user agents to access local mailboxes using the access protocol.

Simple Mail Transfer Protocol

SMTP runs on port 25 with TCP.

The standard message format has some serious limitations:

  • No encryption.
    • SMTPS adds encryption by enabling TLS.
  • Only ASCII characters.
  • Only supports text format.

Multipurpose Internet Mail Extensions (MIME) specification defines useful extensions to SMTP:

  • text/plain: normal ASCII message.
  • text/html: HTML formatted message.
  • image/jpeg: contains only a JPEG image.
  • multipart/mixed: contains multiple media parts.

Example of SMTP message:

HELO example.com
MAIL FROM: imperialcollege@example.com
RCPT TO: email@ic.ac.uk
DATA From: Imperial College <imperialcollege@example.com>
To: Test User <email@ic.ac.uk>
Subject: This is the subject!

This is the body!
.
QUIT

SMTP Headers

  • To: e-mail address(es) main destination
  • Cc: e-mail address(es) to send copies
  • Bcc: e-mail address(es) to send blind copies
  • From: name of sender(s)
  • Sender: e-mail address sender
  • Received: line added by each intermediate transfer agent
  • Return-path: return address
  • Date: the date and time the message was sent
  • Subject: short summary of the message
  • Reply-To: email address to which replies should be sent

The Sender field is often the same as From and Sender so can be left out.

Structure of SMTP Message

  • HELO line
    • HELO
    • Space
    • Host name
  • Header lines (optional)
    • Header name
    • Space
    • Header value
  • Empty line
  • Message body
  • A single dot .
  • QUIT line

SMTP client checks if a line of message body starts with a dot, and inserts another dot in front. SMTP server checks if a line of message body starts with a dot and another character, and deletes the first dot.

Post Office Protocol

SMTP assumes that the user agent and transfer agent run on the same host. POP3 allows a user to access the host mailbox from a different machine.

POP3 runs on port 110 with TCP.

Example of POP3 message:

+OK POP3 server ready
USER kgk
+OK
PASS longcatislong
+OK login successful
LIST
1 2505
.
RETR 1 (pops/sends message #1)
DELE 1
QUIT
+OK POP3 server disconnecting

Internet Message Access Protocol

POP3 deletes retrieves message from the host by default. IMAP stores mail on the host server so that it can be accessed by multiple machines and users.

File Transfer

File Transfer Protocol

https://www.jscape.com/blog/active-v-s-passive-ftp-simplified

FTP enables the exchange of files between a client and a server.

  • Uses two connections to transfer files:
    • Control connection is used to send commands and responses.
    • Data connection is used to transfer the actual files.
  • Enables more reliable and efficient data transfer
  • Active mode
    • Client establishes the control connection.
    • Server establishes the data connection (i.e. server takes active role).
      • Client port is random, which may be blocked by firewall.
  • Passive mode
    • Client establishes the control connection.
    • Client establishes the data connection (i.e. server takes passive role).
      • Client port is still random, but firewalls usually allow inbound traffic from sessions initiated by the client.
