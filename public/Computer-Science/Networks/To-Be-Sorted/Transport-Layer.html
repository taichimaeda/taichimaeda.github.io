Transport Layer

Basic assumptions for the underlying network layer:

  • every host has one unique IP address.
  • best-effort delivery service.
  • no guarantee on the integrity of packets.
  • no guarantee on the order of packets.

Port Numbers

Each application running on a particular host is identified by a unique port number to distinguish simultaneous connections.

  • 0-1023 are reserved well-known ports.
  • 1024 to 49151 are for any user application to use.
  • 49152 to 65535 are dynamic for clients to use temporarily.
    • Randomly selected as client port when initiating a connection to server port.

Connection can be identified by two pairs of host + port + TCP/UDP.

Transmission Control Protocol

TCP is a connection-oriented protocol that runs on port 80. TCP data are called segments.

Features of TCP:

  • Multiplexing/demultiplexing
    • For application identification.
  • Reliable data transfer
  • integrity check
  • Ordered delivery
  • Connections
  • Streams
  • Congestion control

Pseudocode for TCP client:

client_socket = new Socket()
client_socket.connect(host, port)
client_socket.send(data)
print(client_socket.receive())
client_socket.close()

Pseudocode for TCP server:

server_socket = new Socket()
server_socket.bind(host, port)
server_socket.listen(max_connections)
while is_server_running:
	client_socket = socket.accept()
	print(client_socket.receive())
	client_socket.send(data)
	client_socket.close()
server_socket.close()

Maximum size of a TCP segment is about 65KB, because the total length field of an IP frame is 16 bits. In practice MSS is much smaller than this value.

MSS and MTU

  • Maximum Segment Size (MSS)
    • Maximum amount of data transmitted in a single segment without headers.
    • Typically related to the MTU of the connection to avoid network-level fragmentation.
  • Maximum Transmission Unit (MTU)
    • Largest link-layer frame available to the sender host.
    • Path MTU Discovery (PMTUD)
      • Determines the largest link-layer frame by traversing all links from the sender host to the receiver host.

TCP Headers

  • Source port (16 bits)

  • Destination ports (16 bits)

  • Sequence number (32 bits):

    • Indicates the sequence number of the first byte carried by that segment in .
    • A random ISN is chosen in order to:
      • Prevent packet spoofing and hijacking.
      • Avoid receiving any leftover segments by mistake.
    • e.g. For MSS=1024B, sequence numbers will be 1, 1025, 2049, …
      • Starts at 1 because sequence number is incremented during three-way handshake.

  • Acknowledgement number (32 bits)

    • Indicates the first sequence number not yet seen by the receiver.
    • Receiver can acknowledge multiple packets at once.

  • Receive window (16 bits)

    • Size of the window on the receiver end.

  • Header length (4 bits)

    • Size of the TCP header in 32-bit words.

  • Options field (32 bits):

    • May be used to negotiate protocol parameters.

  • URG flag (1 bit)

    • Used to inform the receiver that the sender has marked some data as urgent.
    • The urgent data are located by the urgent data pointer field.

  • ACK flag (1 bit)

    • Used to signal a valid acknowledgment.

  • PSH flag (1 bit)

    • Used to solicit the receiver to pass the data to the application immediately.

  • RST flag (1 bit)

  • SYN flag (1 bit)

  • FIN flag (1 bit)

  • Checksum (16 bits)

    • Used to detect transmission errors.

  • Urgent pointer (16 bits)

Three-way Handshake of TCP

Establishing a TCP connection:

  1. Client sends a TCP segment with a SYN and client ISN.
  2. Server responds with a TCP segment with a SYN, ACK and server ISN.
  3. Client responds with an ACK.

Four-way Handshake of TCP

Terminating a TCP connection:

  1. Client sends a TCP segment with a FIN.
  2. Server responds with an ACK.
  3. Server sends a TCP segment with a FIN.
  4. Client responds with an ACK.

Packet 2 and 3 are be separate because many APIs allow servers to continue sending packets after receiving FIN request.

Error Detection in TCP

Checksum is added in the TCP header to detect transmission errors (e.g. parity bit; XOR sum of all bits in data).

Reliable Data Transfer in TCP

TCP provides mechanisms to ensure reliability in data transfer. Note that IP in the network layer is a unreliable best-effort protocol.

Sender uses the following rules for segment transmission:

  • No arrival of ACK until timeout:
    • Immediately resend segment.
  • Arrival of ACK with duplicate SN:
    • Ignore ACK and continue transmission.

Receiver uses the following rules for ACK generation:

  • Arrival of segment with invalid checksum (i.e. corrupted):
    • Duplicate ACK: immediately send duplicate ACK for the same segment.
  • Arrival of segment with duplicate SN:
    • Duplicate ACK: immediately send duplicate ACK for the next segment.
  • Arrival of in-order segment with expected SN:
    • If all segments up to expected SN are acknowledged:
      • Delayed ACK: wait for the next segment (up to 500ms) before sending ACK for it.
    • If some segments up to expected SN are not acknowledged:
      • Cumulative ACK: immediately send cumulative ACK for all non-acknowledged segments.
  • Arrival of out-of-order segment:
    • If SN is higher than expected SN (i.e. has a gap):
      • Duplicate ACK: immediately send duplicate ACK for the lower end of gap; keep the out-of-order segment until timeout.
    • If SN is lower than expected SDN (i.e. fills a gap):
      • Immediate ACK: immediately send ACK if segment starts at the lower end of gap.

Sliding Window in TCP

With a sliding window the sender has up to unacknowledged segments in the pipeline.

The sender’s state machine is represented by a queue of acknowledgements.

The receiver window size is sent with the receiver’s first ACK segment. Note that a receiver window size of zero is legal.

Congestion Control in TCP

Congestion means that the queue of one or more routers between the sender and the receiver overflow.

TCP uses the following algorithms for congestion control:

  • Slow Start (SS)
    • Set .
    • Send segments and wait for ACKs for all of them.
    • If congestion occurred:
      • Switch to CA with MD.
    • If :
      • Switch to CA with AI.
    • Otherwise:
      • Double and repeat.
  • Congestion Avoidance (CA)
    • Send segments and wait for ACKs for all of them.
    • If congestion occurred:
      • CA with multiplicative decrease (MD).
      • Halve and repeat.
      • If :
        • Switch to SS.
    • Otherwise:
      • CA with additive increase (AI).
      • Increment by and repeat.
  • Fast Recovery
    • In SS and CA with Fast Recovery, the following rules are used instead of checking and congestion:
      • If timeout occurred:
        • Set and switch to SS.
      • If 3 duplicate NACKs received:
        • Halve and switch to CA with AI, skipping SS.
  • Timeouts
    • Use .
    • Timeout interval must be:
      • Larger than RTT to avoid unnecessary retransmissions
      • Not too far from RTT to detect lost segments as soon as possible.

TCP congestion protocol can be configured per device.

Wireless TCP

When packets are lost, TCP slows down because it assumes there is congestion.

However in wireless environments, it should attempt more retransmissions because packets usually get lost due channel reliability issues:

  • Split wireless connections into mostly wired and some wireless connections
  • Let the base station try retransmissions without informing the sender.

User Datagram Protocol

UDP is a connection-less protocol that runs port 53, 68 etc. UDP data are called datagrams.

  • Multiplexing/demultiplexing
    • For application identification.
  • Integrity check
  • Simplicity
    • No flow control
    • No error control

Some reasons for using UDP over TCP:

  • Finer application level control (e.g. real-time, Skype, etc.)
  • No connection establishment (i.e. faster than TCP).
  • No connection state.
  • Small packet header overhead.

For video-streaming we can either use UDP or TCP with pre-buffer.

Pseudocode for UDP client:

client_socket = new Socket()
client_socket.send(host, port, data)
print(client_socket.receive(host, port))
client_socket.close()

Pseudocode for UDP server:

server_socket = new Socket()
while is_server_running:
	print(server_socket.receive())
	server_socket.send(packet)
server_socket.close()

Maximum size of a UDP datagram is about 65KB because its length field is 16 bits. In practice 500 to 1,000B datagrams are used because the smaller they are, the more likely they are to make it intact!

IP cannot replace UDP because we still need the port number fields to deliver the datagram to the correct application.
