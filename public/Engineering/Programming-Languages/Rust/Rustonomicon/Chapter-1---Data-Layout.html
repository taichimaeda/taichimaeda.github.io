Chapter 1 - Data Layout

Representation

repr(Rust) - The Rustonomicon (rust-lang.org)

Composite types:

  • Structs (named product types)
  • Tuples (anonymous product types)
  • Arrays (homogeneous product types)
  • Enums (sum types - tagged unions)
  • Unions (sum types - untagged unions)

A type’s size must always be a multiple of its alignment

Composite structures have an alignment equal to the maximum of their fields’ alignments.

Memory Alignment allows efficient memory/cache access.

Rust does guarantee that two instances of A have their data laid out in exactly the same way. However Rust does not currently guarantee that an instance of A has the same field ordering or padding as an instance of B.

This is unlike C where the ordering of members in a struct is the same as its definition.

In C this is not an issue because programmers can manually reorder its members to pack them as compactly as possible.

But this is a problem in Rust because of monomorphised and enum types - see examples in repr(Rust) - The Rustonomicon (rust-lang.org)

  • For monomorphised types it is kind of obvious.
  • For enum types this unspecified behaviour allows greater extent of optimisations like:
    • Representing None variant with a null pointer (without a tag).
    • Flattening the tags of nested enum types.

Exotically Sized Types

Exotically Sized Types - The Rustonomicon (rust-lang.org)

Dynamically Sized Types

Rust supports Dynamically Sized Types (DSTs): types without a statically known size or alignment. … Because they lack a statically known size, these types can only exist behind a pointer.

Common dynamically sized types (DSTs):

  • Trait objects e.g. dyn MyTrait
  • Slices e.g. [T] and str
  • Any structs with DST as their last field.

DSTs are sometimes known as wide pointers.

The information that completes a trait object pointer is the vtable pointer.

A trait object is composed of:

  • Pointer to data
  • Pointer to vtable, or method-lookup table (MLT)

The information that completes a slice pointer is just the number of elements it points to.

A slice is a view to array or Vec, and is composed of:

  • Pointer to array or Vec, which then points to data
  • The number of elements in the view

Structs can actually store a single DST directly as their last field, but this makes them a DST as well.

Storing a DST at the end of a struct is legal because the offset for each field will remain fixed in machine code, so there will be no performance loss.

Zero Sized Types

Rust also allows types to be specified that occupy no space.

Common zero-sized types (ZSTs):

  • Empty structs e.g. struct Nothing
  • Empty tuples e.g. ()
  • Empty array e.g. [u8; 0]
  • Any structs whose fields are all ZSTs

ZSTs are useful for pattern matching and more.
One of the examples presented in the doc is the implementation of Set with Map:

However in Rust, we can just say that Set<Key> = Map<Key, ()>. Now Rust statically knows that every load and store is useless, and no allocation has any size. The result is that the monomorphized code is basically a custom implementation of a HashSet with none of the overhead that HashMap would have to support values.

Pointers to ZSTs in unsafe code requires some care because they do not exist:

Safe code need not worry about ZSTs, but unsafe code must be careful about the consequence of types with no size. In particular, pointer offsets are no-ops, and allocators typically require a non-zero size.

Empty Types

Rust also enables types to be declared that cannot even be instantiated. These types can only be talked about at the type level, and never at the value level. Empty types can be declared by specifying an enum with no variants:

Empty types are akin to ZSTs in that they do not occupy space, but they can neither be instantiated.

Use of empty types are primarily type-level:

For instance, suppose an API needs to return a Result in general, but a specific case actually is infallible. It’s actually possible to communicate this at the type level by returning a Result<T, Void>. Consumers of the API can confidently unwrap such a Result knowing that it’s statically impossible for this value to be an Err, as this would require providing a value of type Void.

In principle, Rust can exploit empty types to optimise code, not just for static analysis e.g. treating Result<T, Void> as T, but this is not implemented yet.

Other Representations

Attributes of the form repr(*) prepended to structs and enums tell Rust to use alternative data layout for primarily FFI and transmute

  • repr(C) - uses C ABI
  • repr(packed) - uses no padding
  • repr(align(n)) - uses n for alignment
