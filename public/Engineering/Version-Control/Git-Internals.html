Git Internals

Git - Git Objects (git-scm.com)

There are three types of git objects:

  • Blob objects
  • Tree objects
  • Commit objects

Unlike csv or subversion which keeps track of changes, git manages snapshots of tree/blob objects. This makes it significantly faster to switch or merge branches.

Git calculates the 40-character checksum hash (SHA1) of these objects based on their content
e.g. d670460b4b4aece5915caf5c68d12f560a9fe3e4.

Tip

You can often just specify the first 5-6 characters of the hash in the CLI, as long as it is unambiguous.

These objects are named after their hash and stored under .git/objects as zlib file.
For organisational purposes, they are grouped by subdirectory corresponding to the first two characters of their hash e.g. .git/objects/d6/70460b4b4aece5915caf5c68d12f560a9fe3e4.

Warning

Git objects with the same content e.g. blob objects with the same content will have the same hash and will be stored in the same file under .git/objects.

Blob Objects

A blob object represents a single file in the working tree.

The content of a blob object is simply the file content e.g.

Note that because the hash of a git object is based on their content only, two files with different file names but with the same content will get the same hash and thus correspond to the same blob object.

Tree Objects

A tree object represents a single directory in the working tree.

The content of a tree object looks like:

100644 blob a906cb2a4a904a152e80877d4088654daad0c859      README
100644 blob 8f94139338f9404f26296befa88755fc2598c289      Rakefile
040000 tree 99f1a6d12cb4b6f19c8655fca46c3ecf317074e0      lib

Tip

This is similar to how inode-based file systems store files.

  • The first column represent the mode of each file/directory.
    • This is similar to the output of ls -l.
    • The first digit represents directory (0) or file (1)
    • The second digit represents symlink (1) or not (0)
    • The remaining digit represents permission of each group i.e. 4 for write, 2 for read and 1 for execute.
  • The second column is the hash of each file/directory.
  • The last column is the name of each file/directory.

Note that modifying a file and committing it will necessarily update all of its parent tree objects.
Also, renaming a file/directory changes the corresponding tree object, not the blob object.

Commit Objects

A commit object represents a single commit.

The content of a commit object looks like:

tree d8329fc1cc938780ffdd9f94e0d364e0ea74f579
parent ca70291031230dde40264d62b6e8d2424e2c9366
author Scott Chacon <schacon@gmail.com> 1243040974 -0700
committer Scott Chacon <schacon@gmail.com> 1243040974 -0700

This is a commit message.

This is straightforward. A commit object points to the parent commit object, the tree object corresponding to the working tree, plus meta info and commit message.

A commit history is the graph formed by these commit objects and the references to their parents.

Info

To visualise, you can think of a DAG of trees of tree/blob objects, where these trees are overlapping with each. Git is a snapshot-based version control system, but this overlap makes it significantly space-efficient.

Note that because the commit message is part of a commit object, editing a commit message will change the commit object hash and thus any child commits, even if the working tree of the commit has not changed at all.

Branches and Tags

Branches and tags are simply pointers to a commit object.

Internally, branches are stored as text files (not zlib because the size is small) under .git/refs/heads, and tags under .git/refs/tags, with the file name corresponding to their branch/tag name.

The content of such file is simply the hash of the commit object they point to:

c6af3bdd357f57a9205d0ec75b350fdd553d5a940

The difference between branches and tags is that tags are static once created, while branches are updated to the latest commit when a new commit is created after HEAD.

Info

These tags directly pointing to a commit object are called lightweight tags.

Alternatively you can create an annotated tag which will point to a tag object instead. This tag object will contain the reference to the commit object and other meta information i.e. annotations.

Remote tracking branches are named like <remote>/<branch>, and are stored under .git/refs/remotes e.g. .git/refs/remotes/origin/main for origin/main. Similarly for tags.

HEAD file

The HEAD file is a text file which stores information about which commit is currently checked out.

The content of the HEAD file looks like:

ref: refs/heads/main

If the main branch is checked out. If the head is detached (i.e. not on any branch), it may look like this instead:

9388f3aca7f5f003cfd0e1b7b5f7dffd2331d432

index File

The index file represents the index or staging area of a git repository.

The index file keeps the list of tree/blob objects that have changed from the working tree of HEAD, and its content looks pretty much like a tree object:

100644 c6af3bdd357f57a9205d0ec75b350fdd553d5a940	contests/abc341/d/test/sample-3.out
100644 987ee7a91eb97f45b5354952279b71c7699a8d180	requirements.txt
100644 9b2eb6974ba74fead19b506d9cba34e1f3d7d3be0	temp.json

The only different is that the last column stores the file path instead of file name.

config File

The config file is an .ini file that simply stores the local config of a git repository, which looks like:

The config file also stores information like the remote repository and upstream branch you would set with git push --set-upstream.

Note that this config is per repository - the global config can be found at ~/.gitconfig.

logs Directory

The .git/logs directory contains the history of the commit objects pointed by pointers like HEAD, main and origin/main.

In addition to the commit hashes, these log files also keep a record of the action e.g. checkout: moving from main to feature/1.

Commands

git add

For each given file/directory, this command creates a new tree/blob object if the file/directory has changed from the working tree in the HEAD, and records this changed object in the index file.

It may reuse objects if already found under .git/objects.

git commit

This command checks the index file updated by git add, and creates a new commit object based on this index file, with the HEAD as its parent.

It recursively traverses the working tree of the HEAD from the root tree object, and replaces the current tree object with a new one if it contains a file/directory from the index file.

git commit --amend

This is the same as git commit in that it compares the HEAD with the index file to create a new tree representing the working tree, but the new commit object created with git commit --amend has HEAD~1 as its parent, not HEAD.

Note that this does not modify the existing commit object pointed by HEAD, but rather creates a new commit diverging from the parent of the HEAD.

git fetch

This command simply fetches the git objects from the corresponding upstream branch in the remote repository, and updates the remote tracking branches.

git pull

This is equivalent to git fetch <remote>/<branch> and git rebase <remote>/<branch> or git merge <remote>/<branch>, depending on the config.

You can change this config by e.g. git config pull.rebase true.

git checkout

This changes the HEAD to the branch or a specific commit provided in the argument, and updates the working tree to that of the checked out commit.

Confusingly, this command is overloaded with many other uses, but these days most of them have an equivalent, domain-specific command like:

  • git restore for restoring a file
  • git switch for switching the branch

git reset

This moves the current branch so that it points to the given commit, and also clears the index by the default --mixed option.

If you git reset to a parent commit, this effectively resets all the changes made after that commit, but you can also git reset to a commit that is not a parent of the HEAD.

git reset has three options which have very, very semantic naming:

  • git reset --hard
    • Moves the branch, clears the index and updates the working tree.
  • git reset --mixed (default)
    • Moves the branch and clears the index, but keeps the working tree.
  • git reset --soft
    • Only moves the branch, but keeps the index and the working tree.
    • The index is updated so that it represents the changes from the given commit, instead of the HEAD.

This is a bit similar to git checkout in that they both update the working tree, but the difference is that git checkout changes the HEAD whereas git reset changes the branch the HEAD points to.

git merge

git merge <branch> merges <branch> into the current branch, where <branch> is typically a feature branch.

It creates a new merge commit H with the two parents C and G, and updates the current branch:

  A---B---C (<branch>)
 /
D---E---F---G (HEAD - main)

  A---B---C (<branch>)
 /         \
D---E---F---G---H (HEAD - main)

Unless the G can be fast-forward to C:

			  A---B---C (<branch>)
			 /
D---E---F---G (HEAD - main)

By default, git merge uses three-way merge algorithm called ort, which roughly works like:

git merges changes, not files

  1. First, find the common base of target and source branches i.e. the HEAD and <branch>.
    1. Or put more precisely, pick the LCA of target and source commits.
  2. Identify the changes between the base and target.
  3. Similarly, identify the changes between the base and source.
  4. Finally, merge these changes with:
    1. Chunks that have not changed in neither target or source.
    2. Chunks that have changed in either target or source, but not both.
    3. Chunks that have changed in both target and source in the same way.
    4. Placeholders for chunks that have changed in both target and source in different ways, to be resolved by the user manually.
      1. This can be overridden to prefer either target or source by specifying the option for the ort algorithm.

git rebase

git rebase <branch> rebases the current branch onto <branch>, where <branch> is typically the main branch.

It finds the common base of the given commit and the HEAD, and creates a linear commit history by appending/replaying all the commits between the base and the HEAD after the given commit:

	  A---B---C (HEAD - feature)
	 /
D---E---F---G (<branch>)

			  A'--B'--C' (HEAD - feature)
			 /
D---E---F---G (<branch>)

These replayed commits are different from the original commits because they contain a different hash for their parent commits.

Replaying the commits involves merging two working trees, just like git merge, so you can specify the same set of strategies for git rebase. But in git rebase this can get really inconvenient because you could get a conflict in each of such commits appended.

Another way to think about git rebase is that it is just git cherry-pick but repeated for all the original commits to be replayed.

git revert

How git cherry-pick and revert use 3-way merge (jvns.ca)

git revert <commit> creates a new commit that reverts the changes brought by the commit <commit> from its parent commit.

To revert commit Y, git uses the three-way merge algorithm with Y as the base, Z as the target and X as the source, so that the changes from the base Y to the source X gets brought into the target Z:

X---Y---Z (HEAD - main)

X---Y---Z---Y' (HEAD - main)

git cherry-pick

How git cherry-pick and revert use 3-way merge (jvns.ca)

git cherry-pick <commit> creates a new commit that brings the same changes brought by the commit <commit> from its parent commit.

To cherry-pick commit Y onto the current branch, git uses the three-way merge algorithm with X as the base, Y as the source and B as the target, so that the changes from the base X to the source Y gets brought into the target B:

A---B (HEAD - main)
 \
  X---Y---Z (feature)

A---B---Y' (HEAD - main)
 \
  X---Y---Z (feature)

git stash

git stashの仕組みが面白い - 鯨飲馬食 (hatenablog.com)

This creates a temporary commit objects W and I for stashing the WIP working tree and the index. The commit W is for storing the working tree, and I for storing the index.

X---Y (HEAD - feature)

    (HEAD - feature)
X---Y---W (WIP on feature)
     \ /
      I (Index on feature)

When you git stash pop or git stash apply, git performs the following steps (or equivalent):

  • git reset --hard the current branch to W.
  • git reset --mixed the current branch to Y, keeping the working tree of W.
  • git reset --mixed the current branch to I.
  • git reset --soft the current branch to Y, keeping the index of I.

The reference to W is stored as text file under .git/refs/stash. I is simply found by following the parent pointer of W. When there are multiple such WIP commits W, git checks the reflog of .git/refs/stash, which contains the history of the contents of the ref stash.
